# /plan：修复结构化分析 JSON 截断导致的解析失败（自动重试 + 输出限额）

## Summary
当前结构化分析偶发报错 `Unexpected end-of-input: expected close marker for Array`，本质是模型返回的 JSON 被截断/不完整（数组/对象未闭合），后端 `ObjectMapper.readTree` 解析失败后通过 SSE `error` 返回，前端以红色提示框展示。目标是：
- 降低 JSON 变长被截断的概率（限制输出规模、压缩字段长度）；
- 当仍出现不完整 JSON 时，后端自动重试一次（使用更严格、更短的提示词）；
- 保持现有数据结构与前端展示不变。

## Current State Analysis（基于仓库现状）
- 结构化分析生成路径：`AnalysisService.generateAndSaveStructuredResult` 调 `AiContractAnalyst.analyzeContract(...)` 获取 JSON → `mapJsonToAnalysisResult(...)` 用 `ObjectMapper.readTree(...)` 解析并落库。解析失败会一路抛到 `handleStreamingAnalysis`，最终 `sendError(...)`。
  - 代码：[AnalysisService.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/AnalysisService.java)
- `JsonSanitizer.extractJsonObject` 只能去掉 fence 并裁剪 `{...}` 范围，无法修复“缺少 ]/}”的截断 JSON。
  - 代码：[JsonSanitizer.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/util/JsonSanitizer.java)
- 结构化分析提示词是硬编码 `@SystemMessage`，目前没有对风险条目数、字段长度做硬性上限，输出较长时更易在超时/输出限制下被截断。
  - 代码：[AiContractAnalyst.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/ai/AiContractAnalyst.java)
- 用户偏好（已确认）：
  - JSON 解析失败时：自动重试一次（推荐）
  - 每个视角风险条目上限：最多 5 条

## Proposed Changes

### 1) 提示词：限制结构化输出规模，降低截断概率
文件：[AiContractAnalyst.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/ai/AiContractAnalyst.java)
- 在 `analyzeContract` 的“分析要求/数组要求”补充强约束：
  - `party_lessor_risks` / `party_tenant_risks`：每个数组最多 5 条，按优先级排序（风险等级/影响/概率/争议成本/修改成本综合）。
  - `clause_text` 长度上限（例如 200–300 字），超出必须截断并用省略号表示。
  - `risk_description` / `legal_basis` / `suggestion` 提醒“简洁、可落地”，避免长段落。
  - `suggestions`：最多 5 条，必须为字符串数组且不得包含 null（已有要求保留）。
- 说明：不改变 JSON key，仅约束内容规模。

### 2) 自动重试：JSON 解析失败后用“更短提示词”再请求一次
文件：
- [AiContractAnalyst.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/ai/AiContractAnalyst.java)
- [AnalysisService.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/AnalysisService.java)

实现方式（不引入 JSON 自动修复，避免误解析）：
- 在 `AiContractAnalyst` 新增一个第二方法（例如 `analyzeContractConcise(...)`），使用更严格的系统提示词：
  - 明确“上一次输出不完整/无法解析，必须输出完整 JSON，字段更短、条目更少、不得输出任何解释文字”。
  - 仍输出同一套 key。
- 在 `AnalysisService.generateAndSaveStructuredResult(...)` 中：
  1. 第一次请求：`analyst.analyzeContract(...)` → `mapJsonToAnalysisResult(...)`
  2. 若捕获到 `JsonProcessingException`（或解析阶段异常）：
     - 通过 SSE `status` 发送一次阶段提示（例如 `retrying_analysis`：“解析失败，正在重试生成更精简的结构化结果”）
     - 第二次请求：`analyst.analyzeContractConcise(...)`
     - 再次解析；若仍失败，则抛出异常走现有 `sendError`（维持可观测失败）

### 3) 错误信息可读性（可选，保持最小改动）
文件：[AnalysisService.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/AnalysisService.java)
- 对 JSON 解析失败的错误消息进行更友好的包装：
  - SSE error message 不直接包含超长原始 JSON（避免前端红框里塞几千字符）
  - message 提示“结构化结果解析失败（可能是模型输出截断），已自动重试；仍失败请点击重试”

## Assumptions & Decisions
- 不做“括号配平修 JSON”的自动修复（风险是误解析导致落库脏数据）；采用“重试一次 + 限额输出”的稳妥策略。
- 不改前端解析/展示结构；结构化结果仍走现有 `analysis_results` 字段。

## Verification Steps
- 人工验收（本地跑一次初次分析）：
  - 正常情况下：结构化 JSON 能解析并落库，前端不再出现“Unexpected end-of-input”红框。
  - 构造失败（可通过临时把 `DASHSCOPE_CHAT_TIMEOUT` 调小或 `max-tokens` 调小模拟截断）：后端会自动重试一次，并在 SSE status 中看到重试提示；若仍失败，错误信息更短且可理解。
- 静态检查：IDE diagnostics 无新增错误。
