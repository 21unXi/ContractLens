# Plan：用开源 LightRAG 替换现有 RAG（当前分支实施清单）

## 目标

在你已创建的 Git 仓库与当前分支内，把 ContractLens 现有 RAG（LangChain4j + Chroma + Neo4j 自研融合）替换为 **HKUDS/LightRAG 开源项目**，让后端通过 LightRAG 的服务化能力完成检索（向量 + 图谱/混合），并保持现有“租房合同分析”产品形态不变（前端不直接调用外部服务）。

1. `/api/knowledge/rebuild` 触发 LightRAG 的索引构建/更新（以 `knowledge_docs` 为唯一知识源）
2. 合同分析（结构化 + SSE 对话）检索上下文来自 LightRAG（`retrieved_context` 与可选 `graph_context`）
3. LightRAG 不可用时可降级到现有向量检索（可配置），并明确暴露“降级原因”
4. `/api/knowledge/status` 可观测：展示 LightRAG 健康、工作目录/工作区、topK 等关键参数与探测结果（避免“伪总量”）

## 当前仓库状态（已具备）

- 已有完整的“自研 RAG（Chroma + Neo4j）”链路，可作为 LightRAG 接入失败时的 fallback
- 已有 `/api/knowledge/rebuild`、`/api/knowledge/status` 与前端展示，可复用为 LightRAG 的运维入口

## 你需要做什么（本地环境/运行顺序）

### 1) 准备 LightRAG 服务（推荐以服务方式集成）

- 方式 A（推荐）：安装并运行 LightRAG Server
  - `pip install "lightrag-hku[api]"` 或克隆仓库后 `pip install -e ".[api]"`
  - 复制并配置 `.env`（LLM/Embedding/OpenAI-like base-url 等）
  - 启动：`lightrag-server`
- 方式 B：Docker Compose 运行 LightRAG（更接近生产，但 Windows 上可能需要额外环境）

### 2) 本地配置（避免提交敏感信息）

- **不要**把 LightRAG/LLM/Embedding/MySQL 等敏感值写回仓库并提交
- 推荐做法（二选一）：
  - A. 使用 `application-local.yml`（加入 `.gitignore`）覆盖：MySQL 密码、Neo4j 密码、LLM key
  - B. 使用环境变量覆盖（如 `NEO4J_PASSWORD`、`SPRING_DATASOURCE_PASSWORD`、`LANGCHAIN4J_OPEN_AI_CHAT_MODEL_API_KEY` 等）

### 3) 验证闭环（手动验收）

1. 调用 `GET /api/knowledge/status`：确认能拿到 `knowledgeDocsCount` 与 probe 的解释字段
2. 调用 `POST /api/knowledge/rebuild`
3. 再次 `GET /api/knowledge/status`
   - LightRAG probe：returnedChunks/docs > 0（并携带 topK/mode/probeQuery）
   - LightRAG 不可用时：状态返回清晰 error，且（若开启 fallback）会标明已降级到旧 RAG
4. 上传租房合同并触发分析（结构化或 SSE）
   - DB：`analysis_results.retrieved_context` 非空
   - DB：`analysis_results.graph_context`：若 LightRAG 提供图谱相关上下文则写入；不可用时明确包含 FAILED 与降级原因

## 验收标准（对齐“真实可解释”）

- `status` 中的 probe 数值都明确为“返回条数（受 topK 限制）”，并携带 topK/minScore/probeQuery
- LightRAG 不可用时：返回清晰 error；若启用 fallback，明确标记“已降级到 legacy RAG”
- 后端不暴露/不提交任何敏感配置（API Key、密码等）
