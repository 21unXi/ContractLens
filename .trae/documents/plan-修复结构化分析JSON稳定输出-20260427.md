# /plan：从源头修复结构化分析 JSON 无法解析（截断/非结构化输出）并保证 SSE 正常工作

## Summary
当前初次分析在“结构化风险分析”阶段反复出现两次生成都无法解析的情况（`JsonEOFException`/`Unexpected end-of-input`），导致 SSE 返回 `error`，并在服务端出现 `response committed already` 相关异常日志。目标是从源头让流程稳定：
- 建立会话 → 检索 → 结构化生成 → JSON 解析落库 → 前端展示 全链路稳定；
- 重点解决“模型输出不完整 JSON / 非严格 schema”的根因，而不是靠低质量降级结果掩盖问题。

## Current State Analysis（已在仓库中核对）
- 结构化阶段链路：
  - `AiContractAnalyst.analyzeContract(...)` 返回 JSON 字符串
  - `AnalysisService.mapJsonToAnalysisResult(...)` 用 `ObjectMapper.readTree(...)` 解析并落库
  - 解析失败会触发一次 `analyzeContractConcise(...)` 重试，仍失败则 `sendError(...)` 发 SSE error
  - 关键代码：[AnalysisService.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/AnalysisService.java)
- LightRAG 模式下，`retrieved_context` 直接使用 LightRAG 返回的 `context`，目前没有总长度限制；当上下文很长时，模型更容易输出超长内容并被截断，导致 JSON 不闭合。
  - 关键代码：[AnalysisService.retrieveContext](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/AnalysisService.java#L349-L368)、[LightRagClient.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/lightrag/LightRagClient.java#L36-L83)
- 当前提示词里的“输出格式示例”包含 `...` / `[...]` 之类的占位写法，虽然写了“不要照抄”，但这类示例容易诱发模型输出非严格 JSON（或在输出时不严格遵守 schema）。
  - 关键代码：[AiContractAnalyst.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/ai/AiContractAnalyst.java)
- SSE error 发送后调用 `emitter.completeWithError(new RuntimeException(...))`，会引发 `response committed already` 相关 Servlet 日志（因为 text/event-stream 已写出）。
  - 关键代码：[AnalysisService.sendError](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/AnalysisService.java#L689-L701)
- 模型参数（Spring starter 可配置项）支持 `maxTokens/timeout/temperature/responseFormat/strictJsonSchema/maxRetries` 等字段；当前希望把 max-tokens 默认回到 2048 以降低“差几个括号闭合却被截断”的概率。
  - 证据：`ChatModelProperties` 含上述字段（来自本地 jar 反编译输出）。

## Decisions（已确认）
- 重点解决“为什么解析不成功并解决”，不再引入“最小可用结构化结果”降级链路。
- 结构化分析默认 `max-tokens = 2048`。

## Proposed Changes（按优先级）

### 1) 缩短/限额上下文（核心：降低截断概率）
文件：
- [AnalysisService.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/AnalysisService.java)

改动：
- 对 LightRAG 的 `retrieved_context` 做总长度限制（例如 6k–10k 字符范围，可配置），与 legacy 检索保持一致的“分段+总量上限”策略。
- 对 `graph_context` 同样做上限（例如 2k–4k 字符范围，可配置），避免图谱/上下文过长。
- 重试（concise）时进一步收紧上下文上限（例如 retrieved_context 3k，graph_context 1k），确保“重试真的更短”。

### 2) 提示词改为“严格 JSON schema 描述”，移除含 `...` 的示例
文件：
- [AiContractAnalyst.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/ai/AiContractAnalyst.java)

改动：
- 将“输出格式示例”替换为**可直接 parse 的 JSON 模板**（不含 `...`），例如用空数组 `[]`、空字符串 `""` 做占位。
- 明确约束：
  - `risk_level` 只能是 高/中/低
  - `party_*_risks` 必须是对象数组（不允许字符串数组）
  - 每个字段长度上限（clause_text、risk_description、legal_basis、suggestion）
  - 输出前必须进行“JSON 自检”（在输出前确认括号闭合、结构可解析）

### 3) 移除“最小结果”路径，失败即失败但 SSE 不抛 Servlet 异常
文件：
- [AiContractAnalyst.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/ai/AiContractAnalyst.java)
- [AnalysisService.java](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/java/com/contractlens/service/AnalysisService.java)

改动：
- 删除 `analyzeContractMinimal(...)` 及其调用链（若仍存在）。
- `sendError(...)`：保留 SSE `error` 事件，但结束连接时改用 `emitter.complete()`（不再 `completeWithError`），避免 `response committed already` 级联异常。

### 4) 配置项对齐（maxTokens/responseFormat/strictJsonSchema）
文件：
- [application.yml](file:///e:/code/repository/ContractLens/contractlens-backend/src/main/resources/application.yml)

改动：
- 默认 `langchain4j.open-ai.chat-model.max-tokens` 设为 2048（仍可用环境变量覆盖）。
- 保持 `response-format: json_object`，并评估开启 `strict-json-schema: true`：
  - 若 DashScope 兼容协议不支持该字段，则仅保留 `response-format`（以实际请求结果为准）。

## Verification Steps
- 选择同一份合同连续触发初次分析 3 次：
  - SSE 必须稳定结束（成功 `done` 或失败 `error`），且不再出现 servlet `response committed already` 异常日志。
  - 成功场景下：结构化 JSON 能稳定解析落库，不出现 `JsonEOFException`。
  - retrying_analysis 触发频率显著降低（上下文限额生效后应明显改善）。
- 静态检查：IDE diagnostics 无新增错误。
