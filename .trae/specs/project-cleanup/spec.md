# Project Cleanup Spec

## Why

当前仓库已经完成主要功能迭代，但随着开发推进，可能遗留无关文件、临时脚本、未使用代码与依赖，导致：

- 构建/启动变慢、体积变大
- 阅读与维护成本升高
- 安全与合规风险增加（例如误提交日志/导出数据）

本次工作目标是做一次“收尾式清理”，在不改变现有业务功能的前提下，让仓库更干净、更可维护、更可交付。

## What Changes

- 清理无关文件/目录（构建产物、临时文件、重复或废弃内容、示例/草稿等）
- 清理无用代码（不可达代码、未被引用的类/方法、无效配置、废弃 API/DTO）
- 清理无用依赖（前后端的未使用依赖、重复依赖、只用于调试但已不需要的依赖）
- 规范化忽略规则（补齐 `.gitignore` 等，避免再次引入产物/临时文件）
- 更新 `PROGRESS.md` ,把里面被清理的内容更新，做一次整理，

## Non-Goals

- 不做功能重构或架构重写
- 不引入新技术栈或新模块
- 不做业务逻辑变化（除非是删除已确认未被使用的路径/接口）
- 不新增线上能力（签约/支付/社交/移动端等）

## Impact

- Affected areas: 后端 Spring Boot 工程、前端 Vue 工程、仓库根目录工程化文件（忽略规则/脚本）
- Risk: 清理误删被间接依赖的文件或配置，可能导致构建失败或运行期缺失资源
- Mitigation: 以“可验证”为前提，任何删除都需通过构建与关键流程回归

## ADDED Requirements

### Requirement: 仓库垃圾清理

系统 SHALL 删除或移除明确无用的仓库文件/目录，同时不影响构建与运行。

#### Scenario: 构建产物与临时文件

- **WHEN** 仓库存在 `target/`, `dist/`, `build/`, `*.log`, `*.tmp`, `.DS_Store` 等非源代码产物/临时文件
- **THEN** 应从仓库中移除
- **AND** 通过忽略规则防止再次被纳入版本管理

#### Scenario: 无关内容

- **WHEN** 仓库存在与“租房合同分析”无关的样例数据、调试输出、临时导出、实验性脚本
- **THEN** 应删除或迁移到更合适的位置（如明确的 `docs/` 或 `scripts/` 目录）
- **AND** 迁移后需保证引用路径与文档一致

### Requirement: 无用代码清理

系统 SHALL 移除已确认未使用或不可达的代码与配置。

#### Scenario: 未被引用的代码

- **WHEN** 类/方法/资源文件在代码路径中无引用且无运行期反射/配置引用
- **THEN** 应删除
- **AND** 删除后需通过编译与关键流程回归

#### Scenario: 废弃接口或 DTO

- **WHEN** 存在已不再被前端调用的接口或与现有返回结构不一致的 DTO
- **THEN** 应删除或合并为唯一实现
- **AND** 需要同步更新前端调用或移除对应调用点

### Requirement: 依赖清理

系统 SHALL 清理前后端未使用依赖，确保依赖图最小化且可复现构建。

#### Scenario: 未使用依赖

- **WHEN** 依赖未被实际引用或仅用于已删除代码路径
- **THEN** 应从构建文件中移除
- **AND** 保持构建通过

## MODIFIED Requirements

无（本次为清理与收尾，不引入新功能需求）。

## REMOVED Requirements

无（本次不改变产品需求范围）。
